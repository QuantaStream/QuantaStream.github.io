Why QuantaStream

SQL semantics, bitmap physical execution.

Analytical workloads often reduce large sets, combine those sets, and only then need concrete values. QuantaStream is built around that shape.

Compressed first

Query bitmap and BSI state directly.

Predicates, joins, and reductions operate over compressed domains before row materialization.

Relationship aware

Translate constraints across tables.

Relationship-vector fields let the planner move candidate sets between parent and child domains.

Familiar surface

Expose a MySQL-compatible path.

The goal is broad driver compatibility while preserving a bitmap-native engine underneath.

QuantaStream treats SQL execution as constraint propagation over compressed bitmap domains.

Architecture

Built around a clean planning and execution boundary.

The planner understands SQL, table roles, join relationships, residual predicates, and late materialization. Data nodes remain shard-aware and focused on bitmap, BSI, and projection work.

Quanta-in-a-Box

Local first, distributed later.

The first release is anchored around a local runtime for development, demos, SQL conformance, and TPC-H experimentation while preserving the future distributed architecture.

Consul and nodes

Metadata, discovery, and shard ownership.

The current distributed model uses Consul for service discovery and schema metadata while nodes own bitmap and BSI-backed shard data.

qsbridge

The new SQL engine center.

Planning, diagnostics, SQL contracts, function registration, and execution boundaries are moving into the qsbridge-based architecture.

Late materialization

Hydrate values only when needed.

The engine favors bitmap algebra, BSI kernels, and relationship-vector reduction before fetching strings or scalar values.

SQL and validation

A practical analytical SQL subset, tested by roadmap suites.

QuantaStream is expanding SQL support through targeted Go tests and SQLRunner suites. TPC-H is the main analytical validation roadmap before formal benchmark positioning.

Analytical query shapes

Filters, projections, joins, grouping, aggregate execution, ordering, limits, offsets, and selected subquery shapes are under active coverage.

Custom bitmap-native functions

Functions such as topn(...) expose operations that naturally map to categorical bitmap cardinality work.

Driver compatibility

The MySQL-compatible protocol path is intended to support common Go, Java, Python, Node.js, and command-line clients.

Roadmap

Preparing a useful core product, not a limited edition.

The first public release is meant to be approachable for local use and serious enough to show why bitmap-native execution matters.

  • 1
    Core 1.0
    Quanta-in-a-Box, SQLRunner validation, schema docs, MySQL-compatible endpoint work, TPC-H-oriented demos, restart/recovery guidance, and contributor-ready tests.
  • 2
    Distributed readiness
    Multi-node deployment hardening, caching strategy, replication/failover workflows, observability, and standards-based authentication and authorization integration.
  • 3
    Future enterprise and multi-region
    Cloud-provider integrations, advanced workload management, analyzer tooling, multi-data-center topology, and deeper Roaring Bitmap/BSI performance work.
